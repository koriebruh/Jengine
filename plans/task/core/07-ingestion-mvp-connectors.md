# Task 07: Ingestion MVP Connectors

## Goal
Implement the three concrete connectors scoped for MVP Phase 0: CSV/Excel file upload, an SFTP poller, and one SWIFT MT940 statement parser. These are the only ingestion sources needed to prove core reconciliation value with design-partner banks (plans/docs/11-scalability-roadmap.md §12.2 Phase 0) and are the first real implementations of the `SourceConnector` interface task 06 defined.

## Prerequisites
Task 06 (ingestion connector framework — the `SourceConnector` interface, registry, and pipeline these connectors plug into must exist first).

## Scope / Deliverables
- `internal/ingestion/connector/csvupload/` — CSV/Excel upload connector (`connector.go`, streaming row parser).
- `internal/ingestion/connector/sftp/` — SFTP poller connector (`connector.go`, polling scheduler, optional inline PGP decrypt hook).
- `internal/ingestion/parsers/mt940/` — MT940 format parser (`parser.go`, per-bank dialect mapping table support), consumed by an SFTP- or upload-triggered connector instance configured with `SourceFormat: "MT940"`.
- Registration wiring: each connector type registered against task 06's `Registry` from `cmd/ingestion-gateway/main.go`.
- Conformance test suite per connector against anonymized sample fixture files (`internal/ingestion/parsers/mt940/testdata/*.sta` or similar, `internal/ingestion/connector/csvupload/testdata/*.csv`).

## Design Reference
- plans/docs/02-data-ingestion.md §3.1 (connector architecture — SFTP/FTP poller "incl. inline PGP decrypt", CSV/Excel/fixed-width upload "streaming parser, no full-file memory load", SWIFT MT940/MT942/MT101/MT103 "batch, per-bank dialect mapping table" — this task implements MT940 only, the other MT dialects are V1 per roadmap scope below).
- plans/docs/11-scalability-roadmap.md §12.2 Phase 0 ("Connectors: CSV/Excel, SFTP, one MT940 parser") — this is the explicit MVP boundary; do not scope in BAI2, ISO20022 camt.053/054, REST/API pull, Kafka topic ingestion, webhook receiver, or JDBC/DB pull connectors, all of which plans/docs/02-data-ingestion.md §3.1 lists as eventually-built but are V1 (task 18+/25) work.
- plans/docs/15-end-to-end-flows.md §15.1 (the concrete walk-through this task's MT940+SFTP path must satisfy end to end: file checksum dedup-or-correction decision at the connector level, `Statement` row creation with `raw_file_ref`, per-line field mapping, before matching engine tasks take over).
- plans/docs/03-canonical-data-model.md §4.1 (`Statement.checksum`/`raw_file_ref`/`format` fields this task's connectors populate; `format` value for the MT940 path is literally `"MT940"`).

## Implementation Notes
- **CSV/Excel connector** (`csvupload`):
  - `Fetch` reads from an already-uploaded file reference (object storage pointer in `ConnectorConfig.Settings`, e.g. MinIO/S3 key set by whatever upload-handling code stores the file — for MVP, a simple local-filesystem-or-MinIO-read is sufficient; a full HTTP upload-handling API endpoint belongs to task 15, this task only needs to consume a file once it's already stored) using a streaming CSV reader (`encoding/csv`) that emits one `RawRecord` per row without loading the whole file into memory — required per §3.1's explicit no-full-file-memory-load requirement, achievable because `encoding/csv.Reader.Read()` is already line-at-a-time.
  - Excel (`.xlsx`) support: use a streaming-capable Excel library (e.g. `qax-os/excelize`'s streaming reader mode) rather than a whole-workbook-in-memory library — if streaming Excel parsing proves substantially more complex than the task budget allows, a documented, deliberate fallback is acceptable: implement CSV fully streaming, implement Excel via whole-sheet-in-memory parsing bounded by a configurable max-file-size guard (reject files above the guard with a clear quarantine reason) rather than blocking the task on a fully-streaming Excel implementation. State whichever choice is made explicitly in code comments — don't leave it ambiguous.
  - Header-row mapping: the connector itself does NOT do field mapping (that's task 08's DSL, applied at pipeline stage 4) — it emits `RawRecord`s with `Payload` containing the raw row bytes/fields plus enough structure (e.g. a small JSON object of `{header: value}` pairs) that stage 3 (Format Parse, owned partly by this connector's parser logic) can hand off cleanly to stage 4.
  - Duplicate-file detection: compute a file checksum (SHA-256) before processing; compare against `Statement.checksum` for the same `account_id`; per tenant-configurable policy (`ConnectorConfig.Settings`), either quarantine as duplicate or treat as a correction (superseding prior statement) — implement both policy branches, default to "quarantine" if unset (safer default — never silently overwrite financial data).
- **SFTP connector** (`sftp`):
  - Uses `github.com/pkg/sftp` + `golang.org/x/crypto/ssh` for the SFTP/SSH client.
  - Polling: `ConnectorConfig.Schedule` (cron expression) drives when `Fetch` is invoked (the scheduler itself can live in `cmd/ingestion-gateway` as a simple cron-based dispatcher, e.g. `robfig/cron`, invoking `Registry.New(...).Fetch(...)` per due connector — do not build a distributed job-queue scheduler here, that level of infra is the batch matching worker's concern in task 12, not ingestion polling at MVP scale).
  - Remote directory listing + "already processed" tracking via `Checkpoint()`/`Cursor` (task 06's type) — cursor state records the last-seen filename+mtime or a processed-file-hash set, so re-polling the same directory doesn't reprocess unchanged files.
  - Inline PGP decrypt: if `ConnectorConfig.Settings` indicates the file is PGP-encrypted, decrypt using a tenant-scoped private key resolved via Vault-path reference (never inline in config, per §16.3) before handing bytes to the format parser — implement using `github.com/ProtonMail/go-crypto/openpgp` (or equivalent); if no PGP key is configured, skip decryption (plain-file mode is the default and must work standalone without requiring PGP setup for local dev/testing).
  - After fetch, hands the raw file bytes to the MT940 parser (or CSV, if an SFTP-delivered file happens to be CSV-formatted — the connector's job is *transport*, the parser is *format*, kept as two orthogonal concerns per task 06's stage split).
- **MT940 parser** (`internal/ingestion/parsers/mt940`):
  - Parses SWIFT MT940 fixed-field-tag format (`:20:`, `:25:`, `:28C:`, `:60F:`, `:61:`, `:86:`, `:62F:` etc.) into a structured intermediate representation: one parsed statement header (account, statement number, opening/closing balance — feeds `Statement.opening_balance`/`closing_balance`/`period_start`/`period_end`) plus a slice of parsed transaction lines (`:61:`/`:86:` pairs — feeds individual `RawRecord`/eventual `Transaction` rows).
  - **Per-bank dialect mapping table**: different banks vary slightly in `:86:` narrative sub-field conventions and date formats — implement a small dialect-config mechanism (a Go map or tenant/connector-configured table keyed by a `dialect` identifier in `ConnectorConfig.Settings`) that lets the same parser handle minor structural variations without a code change per bank; ship one default/generic dialect plus the capability for tenants to specify a custom one (even if only one concrete alternate dialect fixture exists in tests at this stage — the mechanism must exist, per §3.1's explicit callout of dialect variation).
  - Field mapping DSL (task 08) is the layer that turns `field_61.amount`/`field_61.debit_credit_mark`/`field_86.narrative` (the exact field names shown in plans/docs/02-data-ingestion.md §3.2's YAML example) into canonical `Transaction` fields — this task's parser must expose exactly those field names (`field_61`, `field_86`, sub-fields `amount`/`debit_credit_mark`/`currency`/`value_date`/`narrative`) in its parsed intermediate structure, so task 08's example mapping spec works against this task's output without modification. Getting this field-naming handoff right is the most important cross-task contract in this pair of tasks.
  - Sign handling: MT940 `:61:` debit/credit mark (`D`/`C`) determines transaction sign — the parser surfaces the raw mark; actual sign application (`apply_sign_from` transform) happens in task 08's mapping layer, not here — this task only needs to expose the raw mark faithfully.
- Registration: in `cmd/ingestion-gateway/main.go`, explicitly register `"csv"`, `"sftp"`, `"mt940"` (or `"sftp_mt940"` if MT940-over-SFTP is modeled as one composed connector type — a reasonable, explicit choice: since MT940 is always delivered via SFTP in the fixture flow, register a composed `sftp_mt940` connector type that internally uses the SFTP transport + MT940 parser, while still keeping the transport and parser as separately testable/reusable Go packages) against task 06's registry constructors.

## Non-Goals / Guardrails
- Do not implement BAI2, ISO20022 (camt.053/054), REST/API pull, Kafka topic ingestion, webhook receiver, or JDBC/DB-pull connectors — all deferred to V1 (task 18+/25) per the explicit Phase 0 scope boundary in plans/docs/11-scalability-roadmap.md §12.2. If you find yourself writing a generic "format" abstraction that would make adding these trivial, that's fine as an emergent side effect, but do not spend task budget building or testing them now.
- Do not implement MT942/MT101/MT103 parsing — only MT940 is in MVP scope; the interface/dialect mechanism may be structured to make adding them easier later, but do not implement or test them now.
- Do not implement the field-mapping DSL engine itself (transform functions, YAML spec parsing) — task 08. This task's parser output must be *shaped* to work with task 08's DSL, but the DSL engine is not built here.
- Do not implement WASM sandboxing, the Connector SDK scaffold CLI, or any third-party-connector certification process — V1/task 25.
- Do not build a general-purpose distributed job scheduler for connector polling — a simple cron-based dispatcher in `cmd/ingestion-gateway` is sufficient at MVP scale; KEDA/Kafka-lag-based autoscaling of ingestion pods is a deployment-topology/V1 scaling concern (plans/docs/11-scalability-roadmap.md §12.1), not something this task implements in application code.
- Do not implement real Object Lock/WORM-compliant object storage writes for the raw file archive — local MinIO mocks this per task 02; this task just needs to write the raw file bytes to the configured object storage bucket and record the resulting `raw_file_ref`.

## Definition of Done
- Conformance tests per connector, run against realistic (anonymized/synthetic) sample files: CSV connector correctly streams N rows without loading the full file into a single in-memory slice (verifiable via a test using a large synthetic fixture and a memory-bounded assertion, or at minimum a structural test proving row-at-a-time channel emission); SFTP connector round-trips against a local SFTP test server (e.g. `testcontainers-go` with an SFTP-capable container, or an in-process `sftp` server test double) including checkpoint/cursor persistence across two poll cycles (second cycle doesn't reprocess unchanged files); MT940 parser correctly parses at least two realistic fixture files (including the two documented `debit_credit_mark` cases and at least one multi-dialect variation) into the exact field-name shape (`field_61.*`, `field_86.*`) task 08 expects.
- Integration test: an end-to-end run of the MT940+SFTP path through task 06's full pipeline (using task 08's mapping DSL once available, or a stub mapping if sequenced before task 08 lands — note the dependency but don't block this task's own tests on task 08's existence, use a minimal internal test mapping if needed) produces correct `Statement`+`Transaction` rows matching plans/docs/15-end-to-end-flows.md §15.1 steps 1-9.
- Duplicate-file test: re-processing an identical file (same checksum) is quarantined-or-treated-as-correction per configured policy, verified both ways.
- Manual verification: `make seed` (wired for real now, building on task 02's placeholder) loads a sample MT940 file through the SFTP+MT940 path into the local dev stack, producing visible `Statement`/`Transaction` rows in Postgres.

## Common Pitfalls
- Scope-creeping into BAI2/ISO20022/webhook connectors because the interface makes them "easy to add while I'm here" — explicitly out of scope; resist the urge, per the Phase 0 boundary.
- Loading the entire CSV/Excel file into memory before emitting any records — defeats §3.1's explicit no-full-file-memory-load requirement and will not scale to the record volumes plans/docs/11-scalability-roadmap.md targets.
- Producing MT940 parser output with field names that don't match task 08's mapping DSL example (`field_61.amount`, `field_61.debit_credit_mark`, `field_86.narrative`) — a mismatch here silently breaks the cross-task contract in a way that might not surface until integration testing; name fields exactly as shown in plans/docs/02-data-ingestion.md §3.2's YAML example.
- Hardcoding a single bank's MT940 quirks into the parser with no dialect abstraction —§3.1 explicitly calls out "per-bank dialect mapping table" as a requirement, not a nice-to-have.
- Storing SFTP credentials or PGP private keys inline in `ConnectorConfig.Settings` — must be a Vault path reference per §16.3's secrets convention, resolved at use-time.
- Building a heavyweight distributed scheduler for SFTP polling — a simple cron dispatcher is the right scope at MVP; don't over-engineer job orchestration that belongs to the batch matching worker's design (task 12) instead.
- Silently overwriting a prior `Statement` on re-upload without an explicit, tested tenant policy branch — both "quarantine" and "treat as correction" must be real, tested code paths, not just one implemented with the other as an unimplemented TODO.
