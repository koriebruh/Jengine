# Task 02: API Client, Data Fetching, and Auth

## Goal
Give every subsequent screen task one shared, typed way to talk to the backend and one shared way to know who's logged in: a `lib/api/` client wrapping the MVP REST API, a TanStack Query provider wired to it with sane defaults (polling-friendly, retry/backoff, optimistic-update support), and OIDC/SSO session handling that forwards the session as a bearer token on every API call. Screen tasks (03–11) consume this layer; none of them should hand-roll `fetch` calls or invent their own auth check.

## Prerequisites
- Frontend task 01 (bootstrap) — needs the project shell, route groups, and the "no global client-state library" convention already in place.
- Core task 04 (tenancy context and routing) — the API's JWT/`tenant_id` claim shape this client must send/parse comes from here.
- Core task 15 (REST API layer, MVP) — this is the actual backend surface being called. **This task cannot be verified end-to-end until core task 15 exists**; until then, build and test against a local mock server (see Implementation Notes) so frontend work isn't blocked on backend sequencing.

## Design Reference
- plans/docs/14-dashboard-frontend.md §14.1 — TanStack Query for server-state/caching/polling/optimistic updates; auth via "same OIDC/SSO session as the API layer... NextAuth.js or a thin custom OIDC client, session cookie forwarded as bearer token."
- plans/docs/14-dashboard-frontend.md §14.3 — `lib/api/` as the one generated-or-shared client location; component/screen code never imports `fetch` directly.
- plans/docs/07-api-extensibility.md §8.1 — REST surface shape this client must match: resource-oriented paths (`/v1/tenants/{id}/accounts`, `/v1/tenants/{id}/breaks/{id}/comments`), `page_token` pagination, `Idempotency-Key` header on mutating requests, OAuth2/OIDC auth mapped to RBAC/ABAC.
- plans/docs/11-scalability-roadmap.md §12.2 Phase 0 — "Basic REST API, no webhooks/GraphQL yet" for MVP.

## ⚠️ Resolved Design Ambiguity: REST now, Connect-RPC later
Two docs describe the API layer differently: plans/docs/07-api-extensibility.md §8.1 and plans/docs/00-overview-and-architecture.md §1.3 describe the **target** architecture as contract-first `.proto` served via Connect-RPC (one definition serving gRPC/gRPC-Web/JSON-REST). plans/docs/11-scalability-roadmap.md §12.2 Phase 0 describes the **MVP** backend as "Basic REST API" with no mention of Connect-RPC. Core task 01's own file (`plans/task/core/01-repo-bootstrap-and-tooling.md`) notes that `.proto` files are introduced "starting with the matching engine (task 10+) and API layer (task 15) tasks" — meaning by the time core task 15 lands, the API *may* already be Connect-RPC-served (giving REST for free via JSON transcoding), or it may still be a hand-rolled `net/http`/chi REST surface with proto used only for internal/event schemas. **This is not yet settled in the core task set at the time this task was written.**

Resolution for this task, so the frontend isn't blocked either way: **build a hand-written, thin TypeScript client (`lib/api/client.ts` + per-resource modules) against plain JSON REST endpoints, matching the resource shapes in plans/docs/07-api-extensibility.md §8.1.** Do not depend on `buf generate`/a Connect-Web codegen step at MVP. This works unmodified whether core task 15 ends up serving plain `net/http` JSON or Connect-RPC's JSON-transcoded REST surface — both are "JSON over HTTP" from the client's point of view. When core's proto/Connect-RPC contracts stabilize (whenever that lands — likely alongside or after task 15), swap the hand-written request/response types for `buf generate`'s TS/Connect-Web output **without changing the TanStack Query hook layer** that screens consume (see Implementation Notes for the seam). Flag this explicitly to whoever picks up that later migration; do not silently let the hand-written types drift as the source of truth indefinitely.

## Scope / Deliverables
- `web/lib/api/client.ts` — low-level fetch wrapper: base URL from env (`NEXT_PUBLIC_API_BASE_URL`), attaches `Authorization: Bearer <token>` from the session, attaches `Idempotency-Key` (UUID v4) on POST/PATCH/PUT/DELETE, JSON parse + typed error shape, handles `401` (redirect to login) and `429` (surface `Retry-After` to caller, per plans/docs/01-multi-tenancy.md §2.4 soft-throttle behavior).
- `web/lib/api/types.ts` — hand-written TS interfaces for the resources needed by MVP screens: `Tenant`, `Account`, `Statement`, `Transaction`, `Break`/`Case`, `CaseComment`, `CaseAuditEvent`, `MatchResult`, `MatchResultLine`, `Connector`/`ConnectorConfig`, paginated-list envelope (`{ items: T[], next_page_token: string | null }`). Field names/types mirror plans/docs/03-canonical-data-model.md §4.1 exactly — do not invent a different casing/shape convention than the backend's documented model.
- `web/lib/api/endpoints/` — one module per resource (`cases.ts`, `matches.ts`, `connectors.ts`, `tenants.ts`) exporting plain async functions (`listBreaks(params)`, `getBreak(id)`, `assignBreak(id, userId)`, etc.) built on `client.ts`. Screen tasks call these, never raw `fetch`.
- `web/lib/query/query-client.tsx` — `QueryClientProvider` with defaults: `staleTime` tuned for polling screens (short, e.g. 5–10s, overridable per-query — frontend tasks 03/05 set explicit `refetchInterval` since MVP has no SSE), `retry` with backoff, global error toast via the `sonner` toast from frontend task 01.
- `web/lib/query/keys.ts` — a query-key factory (`queryKeys.breaks.list(filters)`, `queryKeys.breaks.detail(id)`, etc.) so cache invalidation in mutation `onSuccess` handlers is consistent across screens instead of each task inventing its own key shape.
- `web/lib/auth/` — auth integration. Use NextAuth.js with a generic OIDC provider (Auth0/Okta/Azure AD-compatible config via env vars) **or** a thin custom OIDC client per §14.1 — pick NextAuth.js unless a concrete blocker surfaces (it covers session cookie handling, CSRF, and token refresh out of the box, which a hand-rolled client would have to reimplement). Deliverables: `web/lib/auth/auth-options.ts` (NextAuth config), `web/app/api/auth/[...nextauth]/route.ts`, a `useSession`-based `useCurrentUser()` hook exposing `{ userId, tenantId, roles }` decoded from the session/JWT.
- `web/middleware.ts` — protects the `(dashboard)` route group: unauthenticated requests redirect to `/login`; the `(auth)/login` route itself is added here (`web/app/(auth)/login/page.tsx`), a minimal "sign in with SSO" button, no custom credential form (OIDC SSO is the only path per plans/docs/09-security-compliance.md §10.3's RBAC role list — Tenant Admin/Recon Manager/Analyst/Approver/Auditor/API Integration Role — all human roles authenticate via SSO).
- `web/.env.example` additions: `NEXT_PUBLIC_API_BASE_URL`, `NEXTAUTH_URL`, `NEXTAUTH_SECRET`, `OIDC_ISSUER`, `OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET`.
- A local mock server or MSW (Mock Service Worker) handler set under `web/lib/api/mocks/` covering the endpoints this task's own verification needs (breaks list/detail, connectors list) — used for this task's Definition of Done and reusable by frontend tasks 03/04/06 in early development before core task 15 is live.

## Implementation Notes
- Token flow: NextAuth session holds the OIDC ID token / access token; `client.ts` reads it server-side (in Server Components / Route Handlers) via `auth()` helper, and client-side (in Client Components using TanStack Query hooks) via a small `getAccessToken()` utility that pulls from the NextAuth session hook. Do not store tokens in `localStorage` — rely on NextAuth's httpOnly session cookie.
- `tenant_id` is a claim on the session/JWT, not something the UI lets a user type in — `useCurrentUser().tenantId` is threaded implicitly into every `lib/api/endpoints/*` call's path (`/v1/tenants/{tenantId}/...`), so screen code never manually constructs tenant-scoped URLs; centralize that path-building in `client.ts` or a small `buildTenantPath()` helper.
- The seam for the future Connect-RPC swap: keep `lib/api/endpoints/*.ts` functions' **signatures and return types** stable (they return the shapes in `types.ts`). When a generated Connect-Web client exists, the internals of these functions can call the generated client instead of `client.ts`'s fetch wrapper, without any screen-level code changing. Note this seam with a code comment at the top of `web/lib/api/endpoints/cases.ts` (and siblings) pointing back to this task file.
- Idempotency-Key: generate once per logical user action (e.g. once when a "confirm match" button is clicked, not regenerated on retry) so retries are actually idempotent — store it in the mutation's variables, not regenerated inside the fetch wrapper on every call.
- Pagination: `page_token`-based per §8.1; `lib/api/types.ts`'s list envelope and TanStack Query's `useInfiniteQuery` (for screens that need it) should use `next_page_token` as the `getNextPageParam` source directly — don't invent an offset/limit convention that contradicts the documented API shape.
- Mock server: prefer MSW since it can run both in tests and in the dev server via a service worker, letting frontend tasks 03–07 be developed and demoed before core task 15 is deployed. Seed mock data shapes matching `types.ts` exactly so swapping to the real API later is a config change (base URL / disable MSW), not a data-shape rewrite.

## Non-Goals / Guardrails
- Do not run `buf generate` or add a Connect-Web codegen step in this task — see the resolved ambiguity above. Do not add `@connectrpc/connect-web` or `@bufbuild/protobuf` dependencies yet.
- Do not implement webhook subscription UI, API key management UI, or GraphQL client — those are frontend task 09 (V1) and out of scope for MVP.
- Do not build role-based UI hiding/showing (OPA-aware conditional rendering) — that's explicitly V2 per plans/docs/14-dashboard-frontend.md §14.4 ("refined ABAC-aware UI... rather than showing then erroring" is Phase 2). This task only needs to expose `roles` on the session for later use, not act on them yet beyond the basic authenticated/unauthenticated gate.
- Do not add a custom credential-based login form — SSO/OIDC only, per the design's auth model.
- Do not introduce a global client-state library to hold "the current user" or "the current tenant" — that's exactly what the NextAuth session + TanStack Query cache already provide; a `useCurrentUser()` hook backed by `useSession()` is sufficient.

## Definition of Done
- Against the mock server: a Client Component using a `lib/api/endpoints/cases.ts` function successfully renders a list of mock breaks through TanStack Query (loading → data states both visibly correct).
- `middleware.ts` verified manually: visiting any `(dashboard)` route while logged out redirects to `/login`; completing SSO sign-in (against a test OIDC provider/dev tenant) lands back on the originally requested route.
- A mutation (e.g. mock "assign break") round-trips: optimistic update shows immediately, then reconciles with the (mock) server response; a forced mock error rolls the optimistic update back and surfaces a toast.
- `Idempotency-Key` header is present and stable across a retried request (verify via browser devtools network tab against the mock server).
- 401 from the mock server triggers a redirect to `/login`; 429 with `Retry-After` surfaces a user-visible "try again in Ns" state rather than silently failing.
- QA findings go in the single root-level `QA_REPORT.md`, open items only.

## Common Pitfalls
- Building the Connect-RPC/proto codegen pipeline anyway "since the design doc says that's the target" — this task explicitly defers that; building it now risks a toolchain the MVP backend (task 15) may not even expose yet, and blocks this task on an unresolved core-team decision.
- Storing the access token in `localStorage`/`sessionStorage` for "simplicity" — defeats httpOnly cookie XSS protection NextAuth provides by default.
- Hardcoding a single tenant's ID anywhere in `lib/api/` — every path must derive `tenant_id` from the session, never a constant, even for local dev convenience (use a seeded dev-tenant session instead).
- Skipping the mock-server layer and declaring this task blocked on core task 15 — the task is explicitly designed to be verifiable without a live backend; use MSW instead of waiting.
- Reinventing pagination as offset/limit because it's "simpler to mock" — must match `page_token` per the documented contract, or every later screen's infinite-scroll/pagination code has to be rewritten when the real API lands.
- Putting `roles`-based conditional UI logic into this task "while I'm already touching auth" — explicitly deferred to V2 per §14.4; this task only needs `roles` present on the session object, unused elsewhere yet.
