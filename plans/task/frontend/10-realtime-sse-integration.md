# Task 10: Real-Time SSE Integration (V1)

## Goal
Replace the MVP polling mechanism in the Case Queue (frontend task 03) and Match Review Queue (frontend task 05) with real-time Server-Sent Events, so open breaks and suggested matches update live as `case.events`/`matching.results` Kafka events occur, without the analyst needing to wait for the next poll tick or manually refresh. This is a **retrofit**, not a new screen — the concrete deliverable is removing two specific `refetchInterval` polling mechanisms and replacing them with an SSE subscription that feeds the same TanStack Query cache.

## Prerequisites
- Frontend task 03 (Case Queue) and frontend task 05 (Match Review Queue) — this task modifies both directly; read their existing polling implementation (the `CASE_QUEUE_POLL_INTERVAL_MS`-style constants they were built with) before starting, since this task removes/replaces that exact code path rather than adding a parallel one.
- Frontend task 02 (API client/auth) — the SSE connection needs the same bearer-token/session auth as REST calls; confirm how an `EventSource` (which cannot set custom headers) authenticates — see Implementation Notes.
- Core task 18 (streaming ingestion, Kafka/Redpanda, V1) — this is what makes `normalized.transactions`/downstream topics real in the first place.
- Core task 19 (streaming matching worker, V1) — populates `matching.results.<shard>` (per plans/docs/06-streaming-architecture.md §7.1) with the events the Match Review Queue's SSE channel needs.
- **Core task 21 (webhook system)** owns the backend SSE/WS gateway (resolved during reconciliation — it was originally an unassigned gap). It exposes `POST /v1/tenants/{tenant_id}/events/stream-token` (mints a short-lived, stream-scoped token) and `GET /v1/tenants/{tenant_id}/events/stream` (the SSE endpoint itself, filtered to that tenant's `case.events`/`matching.results` topics). This frontend task consumes that contract; it does not build the gateway itself.

## Scope / Deliverables
- `web/lib/realtime/sse-client.ts` — a small `EventSource`-wrapping client: connects to the tenant's SSE channel, parses incoming events by type (`break.created`, `break.assigned`, `break.sla_warning`, `break.sla_breached`, `break.resolved`, `match.found`, `match.auto_confirmed`, etc. — the same event catalog as the webhook system per plans/docs/07-api-extensibility.md §8.2, reused here for push instead of webhook delivery), handles reconnect-with-backoff on connection drop (SSE/`EventSource` auto-reconnects by default, but add a max-backoff ceiling and a visible "reconnecting..." indicator rather than relying on silent browser defaults).
- `web/lib/realtime/use-sse-subscription.ts` — a React hook (`useSSESubscription(eventTypes, onEvent)`) wrapping `sse-client.ts` with proper connect-on-mount/disconnect-on-unmount lifecycle, usable by any screen (built generically even though only two screens use it in this task, since the gateway/event model is shared infrastructure).
- Modify `web/app/(dashboard)/cases/page.tsx` (frontend task 03) — remove the `refetchInterval` polling constant; wire `useSSESubscription` for `break.*` events to call `queryClient.invalidateQueries` (or, for finer-grained updates, `queryClient.setQueryData` to patch the specific row) on `queryKeys.breaks.list`. Keep a **fallback poll at a much longer interval** (e.g. 2–5 min) as a safety net in case the SSE connection silently drops without the reconnect logic catching it — do not remove polling entirely with zero fallback.
- Modify `web/app/(dashboard)/matches/page.tsx` (frontend task 05) — same retrofit pattern for `match.found`/`match.auto_confirmed`-type events feeding `queryKeys.matches.list`.
- `web/components/realtime/connection-status-indicator.tsx` — small, unobtrusive UI element (e.g. in the topbar or per-screen) showing live/reconnecting/offline SSE connection state — per plans/docs/07-api-extensibility.md's transparency theme, don't hide connection health from the user silently.

## Design Reference
- plans/docs/14-dashboard-frontend.md §14.1 — "SSE chosen over full WebSocket for simplicity (one-directional server→client push is all the UI needs; client actions go back over normal REST/gRPC calls, not over the socket)" — this is the exact architecture: SSE for push, REST for all mutations (confirm/reject/assign/etc. still go through `lib/api/endpoints/*` exactly as before, unchanged by this task).
- plans/docs/14-dashboard-frontend.md §14.2 screens 2 and 4 — "Live-updates via SSE when new breaks open or existing ones change status" (Case Queue) and the Match Review Queue's real-time population as suggested matches are found.
- plans/docs/06-streaming-architecture.md §7.1 — topic table: `case.events.<shard>` (partition key `tenant_id`) and `matching.results.<shard>` (partition key `tenant_id, account_id`) are the Kafka sources the backend gateway subscribes to; this task's frontend client doesn't touch Kafka directly, only the SSE HTTP endpoint the gateway exposes.
- plans/docs/11-scalability-roadmap.md §12.2 Phase 1 — "Frontend: full sellable screen set... SSE-based live updates" confirms this is V1, and plans/docs/14-dashboard-frontend.md §14.4 Phase 0 explicitly says "No SSE — polling is fine at MVP volume," confirming frontend tasks 03/05 were correctly built polling-only and this task is the intended, planned retrofit — not an unplanned rework.

## Implementation Notes
- **Auth for `EventSource`**: the browser's native `EventSource` API cannot set an `Authorization` header. Core task 21 resolves this concretely: call `POST /v1/tenants/{tenant_id}/events/stream-token` first (normal bearer-authenticated REST/Connect-RPC call, sent with a regular header), receive a short-lived, stream-scoped token in the response body, then open `new EventSource('/v1/tenants/{tenant_id}/events/stream?token=' + streamToken)`. Never pass the long-lived REST/session bearer token itself as a query parameter — always mint and use the short-lived stream token for this purpose.
- Per-tenant channel: the SSE endpoint is per-tenant (`tenant_id` already resolved server-side from the session/token, same as REST calls) — the frontend does not need to pass `tenant_id` explicitly beyond what auth already carries.
- Fine-grained cache updates vs. full invalidation: prefer `queryClient.setQueryData` to patch the specific affected row (by ID, from the event payload) into the existing cached list when the event payload carries enough data to do so cleanly; fall back to `invalidateQueries` (triggering a refetch) when the event only signals "something changed, go re-fetch" without a full payload. Confirm which shape the actual `case.events`/`matching.results` payloads carry before choosing per-event-type — don't uniformly refetch-on-every-event if the payload already contains what's needed, since that defeats some of the latency benefit of push in the first place.
- Connection lifecycle: one shared `EventSource` connection per authenticated session (not one per screen/component) is preferable to avoid multiple redundant streams if a user has multiple tabs/screens open simultaneously relying on the same hook — consider a small connection-manager singleton in `sse-client.ts` rather than each `useSSESubscription` call opening its own raw `EventSource`.
- Reconnection UX: exponential backoff with a visible cap (e.g. stop auto-retrying after N attempts and show an explicit "reconnect" button) rather than silently retrying forever with no user feedback.

## Non-Goals / Guardrails
- Do not build the backend SSE/WS gateway itself — that is core-side infrastructure (see the Prerequisites note above); this task only integrates against it. If, when this task is picked up, the gateway genuinely doesn't exist yet in any core task, stop and flag the gap rather than building a frontend-only mock and calling the task done — a fake SSE integration provides no real value and creates false confidence.
- Do not add SSE to any screen other than Case Queue and Match Review Queue — those are the two named in this task's scope and in plans/docs/14-dashboard-frontend.md's live-update descriptions (screens 2 and 4). Connector Monitor, Overview Dashboard, Tenant Admin, Audit Viewer stay on their existing polling (or no polling) conventions unless a future task explicitly extends SSE to them.
- Do not remove the fallback polling entirely — SSE connections can silently degrade in ways a frontend client can't always detect instantly; a long-interval fallback poll is required, not optional, per the Scope/Deliverables note above.
- Do not switch to a full bidirectional WebSocket — the design explicitly chose SSE because client actions stay on REST; do not "upgrade" to WebSocket for perceived symmetry.
- Do not put mutation actions (confirm/reject/assign) over the SSE channel — SSE in this design is strictly server→client push; all client-initiated actions remain REST calls exactly as built in frontend tasks 03/05.

## Definition of Done
- Against a real (or realistically simulated, e.g. via a local SSE test endpoint emitting synthetic events) gateway: opening the Case Queue screen and triggering a `break.created`/`break.assigned` event server-side updates the visible table without a manual refresh or waiting for the fallback poll interval.
- Same verification for the Match Review Queue with `match.found` events.
- Killing the SSE connection (e.g. via devtools network throttling/blocking) triggers the reconnect-with-backoff behavior and the connection-status indicator visibly reflects "reconnecting" then either recovers or reaches the capped-retry "reconnect manually" state.
- Fallback polling still functions (verified by disabling SSE server-side entirely and confirming the screen still eventually reflects new data via the long-interval poll, just with expected added latency).
- Manual verification: two browser tabs open to the Case Queue simultaneously both reflect a live change without either tab needing a manual refresh, and devtools network tab shows a single shared connection strategy is at least considered/documented even if per-tab connections are the simpler shipped implementation (note in `QA_REPORT.md` if per-tab connections were shipped instead of a shared singleton, as a known follow-up, not a silent gap).
- QA issues in the single root-level `QA_REPORT.md`.

## Common Pitfalls
- Building this task against an assumed gateway contract without confirming it's actually been built in a core V1 task — the Prerequisites section flags this explicitly as an unresolved cross-team dependency; verify before investing significant effort.
- Passing a long-lived REST bearer token as an SSE query parameter without considering exposure via logs — prefer a short-lived, stream-scoped token if the gateway supports issuing one.
- Removing polling entirely with no fallback, leaving the screen silently stale if the SSE connection drops without visibly erroring.
- Adding SSE to screens beyond the two named here "since the infrastructure is already there" — scope discipline matters; other screens' real-time needs (if any) are a separate future decision, not an automatic extension of this task.
- Building a full WebSocket implementation "since it's basically the same effort" — contradicts the explicit SSE-over-WebSocket design rationale in §14.1.
- Re-fetching the entire list on every single SSE event when the event payload already contains enough data for a targeted cache patch — wastes the latency benefit SSE was introduced to capture.
